设计模式这系列文章，是最近在考虑工作方向选择什么时候想着要写的文章。由于有好多乱七八糟的内容不方便放在正文里面，所里单独列出来放到一篇文章。

实际上，我写这系列文章的初衷算是记录学习的过程，对于好多设计模式的理解并不一定标准，也并不一定很适合开发利用。 我之前在看书的时候，很是讨厌长篇大论，一堆专业性的文字描述，外行人完全没有心思看下去。 或许有的人说，本身编程开发就不是向大众开放的嘛，没必要写的这么荒诞。 我在刚入门编程的时候，是跟随马士兵老师的视频在自学，那时候挺伤一天也不觉得累。但是在上c语言编程课的时候，就听一会玩一会手机。也不是说老师能力不够，而是在学习的时候总没有一种很活跃的气氛，让人提不起兴趣。 后来，我在学习专业课的时候，总是沉不下去心来听，感觉没有应用的意义。而且老师跟学生之间就是一个师生的关系，没办法进行一种平等的交流。 我理解的比较理想的课堂是老师循循善诱，学生能够靠自己的力量，对学习提起兴趣，主动学习。 我还记得，在看一本书的时候，物理老师教重力加速度，怎么讲的呢？ 那天愚人节，学生用水桶盛满水，捉弄他。但是他提前预判了下，恰好躲过了水！ 然后他就把讲课内容换成了为什么他能够躲开，学生应该怎么设计才能够让水淋到他。 好多东西都是有共性的，很多理论都是能够在生活中找到原型的。如果只能够长篇大论，学习的兴趣就失去了。 在读明朝那些事的时候，我发现，原来历史还可以这么吸引人，原来在严肃的史书中可以演绎出这么多动人的事迹。我就在想，如果我写博客，能够尽力用轻松幽默的语言来表达出一些内容，就很了不起了。然而我只是一个工科狗，文笔不够，幽默感也不强，只希望能够让大家对程序员有点其他的理解就知足啦。 说完了这么多想法，来谈一谈这系列吧。 这系列参考了很多文章，很多人的博客，很多专业书籍。好多都读完索然无味，好多也都让我发笑，很感兴趣。 有很多值得感谢的，只是太多了，记录不完，只好记录印象比较深刻的了。 1.
